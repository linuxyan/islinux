💡 The Solution: Incremental Layer Transfer

Docker images consist of multiple layers, each representing a step in the Dockerfile. The key insight: most layers don't change between versions.

How Docker Layers Work

  • Base layer: Ubuntu, Alpine, or Debian—rarely changes
  • Dependency layer: npm install, pip install, Maven—changes occasionally
  • Application layer: Your source code—changes frequently
  • Config layer: Environment variables, entrypoint—changes rarely
Core Strategy: Instead of transferring the entire image, identify and transfer only the changed layers. If 80% of layers are identical, why transfer them?

🛠️ Introducing ddil: Docker Diff Image Layers

ddil is a command-line tool that compares two Docker images and extracts only the different layers into a compact archive.

What ddil Does

  • ✅ Automatically pulls and parses Docker images
  • ✅ Compares all layers between two versions
  • ✅ Identifies and removes identical layers
  • ✅ Packages different layers into a single tar.gz
  • ✅ Supports both tag comparison and full URL comparison

How It Works

1. docker pull → Fetch both old and new images
2. docker save → Export images to tar format
3. Parse manifest.json → Extract layer lists
4. Compare layer hashes → Identify duplicates
5. Remove identical layers → Keep only differences
6. Compress to diff.tar.gz → Ready for transfer

📖 Getting Started with ddil

Installation

# Clone and build
git clone https://github.com/linuxyan/ddil.git
cd ddil
go build -o ddil

# Verify installation
./ddil --help

Usage Method 1: Tag Comparison (Recommended)

Compare different versions of the same image:

./ddil myapp:v1.0 myapp:v1.1 myapp:v1.2

This compares myapp:v1.0 (base) against myapp:v1.1 (new), outputting the differential layers.

Usage Method 2: Full URL Comparison

Compare images from different registries:

./ddil registry.old.com/myapp:latest registry.new.com/myapp:latest

Understanding the Output

After completion, you'll see:

Output Files

existlayers - List of identical layers that were removed

⚠️ Prerequisites

  • Docker must be installed and accessible via CLI

🚀 CI/CD Integration Guide

GitHub Actions Workflow

name: Deploy with Diff Layers

on:
  push:
    branches: [main]

jobs:
  build-and-deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Build ddil
        run: |
          go build -o ddil

      - name: Generate differential layers
        run: |
          ./ddil myapp:${{ github.event.before }} myapp:${{ github.sha }}

      - name: Upload diff artifact
        uses: actions/upload-artifact@v4
        with:
          name: image-diff
          path: "*.tar.gz"

      - name: Deploy to production
        run: |
          ssh deploy@prod "docker load -i image-diff.tar.gz"

Production Deployment Script

#!/bin/bash
# deploy.sh - Apply differential layers on target

BASE_IMAGE="myapp:v1.0"
DIFF_FILE="myapp__v1.0__v1.1__diff.tar.gz"

# Ensure base image exists
docker pull $BASE_IMAGE

# Load differential layers
docker load -i $DIFF_FILE

# Retag as latest
docker tag myapp:v1.1 myapp:latest

# Restart containers
docker-compose up -d

❓ Frequently Asked Questions

Q: How do I apply the diff file on the target machine?

A: The target machine needs the base image first. Then use docker load -i diff.tar.gz to load the differential layers. Docker recognizes them as complete layers.

Q: What if the base images are completely different?

A: ddil will identify and remove any matching layers. If base images differ entirely, the diff will be large. Consider standardizing base images across versions.

Q: Does ddil work with private registries?

A: Yes. ddil uses the local Docker daemon, so if docker pull works (with authentication), ddil will work.

Q: How much transfer time can I save?

A: Depends on similarity between versions. Code changes typically affect 1-2 layers, reducing transfer by 80-95%. On 100Mbps networks, this means going from 15 minutes to 90 seconds.

Q: Can I use this for multi-stage builds?

A: Absolutely. ddil compares the final image regardless of how it was built. Multi-stage builds often produce smaller diffs since build dependencies aren't included.
